# Mirmaz League — Backend

بنية خادم متزامن حقيقي لـ Mirmaz League. كل ما يمس الاقتصاد أو نتيجة
المباراة يُحسم على الخادم — العميل عارض فقط.

---

## المعمارية

```
Next.js (Vercel)  ──HTTPS──►  Supabase          (Auth, Postgres, Storage, RLS)
      │                            ▲
      └──WebSocket──►  Game Server │            (Colyseus 0.17 على Railway/Fly)
                       service_role ┘
```

**لماذا خادم منفصل عن Supabase؟** Supabase Realtime يبث تغييرات قاعدة
البيانات — ممتاز للإشعارات، غير مناسب لمباراة 60 ثانية تحتاج مؤقتاً
موثوقاً وعشوائية لا يراها العميل. Colyseus يعطي Rooms + matchmaking +
مزامنة حالة جاهزة.


```bash

npm run build && npm start
```


## ثغرات وجدها الاختبار (وأُغلقت)

هذه ليست نظرية — كلها نجحت فعلاً قبل الإصلاح:

1. **`user_cards` كان قابلاً للكتابة** — RLS مفعّل لكن بلا سياسة INSERT.
   أي لاعب يمنح نفسه Super Rare. → سحب صلاحيات + `force row level security`.

2. **`apply_match_result` كان مستدعىً من أي لاعب** — `CREATE FUNCTION`
   يمنح EXECUTE لـ PUBLIC افتراضياً، و `REVOKE` كتبته *قبل*
   `CREATE OR REPLACE` فأُعيد المنح. → REVOKE بعد التعريف.

3. **`guard_profile_economy` خنق `open_pack` نفسه** — التريغر يفحص
   `auth.role()`، لكن `SECURITY DEFINER` يحتفظ بدور *المستدعي*. الدالة
   المخوّلة لتحريك العملات كانت تُمنع بحارسها. → سياق
   `app.economy_ctx` تفتحه الدوال الموثوقة فقط.

## ميكانيك السحب (مُصحَّح)

**كل مركز له 4 صناديق خاصة به** — لا 4 صناديق للتشكيلة كلها.

| الملعب | المراكز | الصناديق لكل لاعب |
|---|---|---|
| ثلاثي | 3 | **12** |
| خماسي | 5 | 20 |
| سباعي | 7 | 28 |
| كامل | 11 | **44** |

القواعد:
- داخل المركز الواحد، اللاعب يفتح الصناديق **بأي ترتيب** ويقرر لكل واحد:
  احتفظ أو ارفض.
- **آخر صندوق يتبقى هو الإجباري** — لا زر رفض. الإجبار صفة *كونه أخيراً*،
  لا رقماً ثابتاً:
  - فتح 1، 2، 3 ورفضهم → **الصندوق 4** إجباري
  - فتح 2، 3، 4 ورفضهم → **الصندوق 1** إجباري
  - فتح 3، 1، 4 ورفضهم → **الصندوق 2** إجباري
- لهذا لا يمكن أن يبقى مركز فارغاً، و"الرفض نهائي" يظل ذا معنى.
- الرفض حتى الصندوق الأخير يفتحه تلقائياً (لا قرار متبقٍ).

الاختبار يغطي **كل الـ24 ترتيباً ممكناً** ويتحقق أن الأخير فقط إجباري.
- **60 ثانية لكل مركز** لا للتشكيلة كلها. انتهاء الوقت يأخذ المكشوف، أو
  يفتح التالي ويأخذه.
- البطاقات تُسحب من كل المجمع بغض النظر عن المركز — صندوق الحارس قد يعطي
  مهاجماً. المركز على البطاقة تجميلي حالياً.

### لماذا هذا ميكانيك أفضل

محاكاة 40 ألف تشكيلة لكل عتبة:

| عتبة القبول | متوسط المجموع |
|---|---|
| 60 (متساهل) | 224.6 |
| 70 | 238.7 |
| **75 (الأمثل)** | **242.0** |
| 80 | 240.7 |
| 90 | 227.9 |
| 95 (جشع) | 219.5 |

يوجد **نقطة مثلى حقيقية**. الجشع الزائد يدفعك للصندوق الإجباري كثيراً،
والتساهل يجعلك تقبل بالضعيف مبكراً. هذا ما يحوّل اللعبة من حظ إلى مهارة —
الميكانيك القديم (4 صناديق إجمالاً) كان يعطي 211.2 بلا قرار حقيقي.

## توازن اللعبة

### معادلة القوة (كانت غائبة من الـ GDD)

```
q      = (base_teacher - 52) / (88 - 52)        # موقع الأستاذ 0..1
rating = tier.min + q × (tier.max - tier.min)
power  = round(rating × 1.15)
```

الصيغة الأولى كانت `base + bonus` مع قصّ للحدود، فثبّتت **75% من
البطاقات البرونزية على نفس التقييم** وتركت `master` بأربع قيم مختلفة
فقط بين 100 أستاذ — أي أن الأستاذ الذي تسحبه لا يهم. التوزيع النسبي
يبقي كل أستاذ متمايزاً داخل كل إصدار.

### منحنى الأهداف (تصحيح للـ GDD)

الـ GDD حدّد جدولاً خطياً ينتهي عند "41-50 = 5:0". محاكاة 200 ألف
مباراة أظهرت أن فروق التقييم تصل 140+، فـ **19.3% من المباريات كانت
تنتهي 5:0** والتعادل شبه مستحيل (1.3%).

```
goals = min(7, floor(sqrt(diff / 3)))
```

| النتيجة | الجدول الخطي | المنحنى |
|---|---|---|
| تعادل | 1.3% | 6.4% |
| 1:0 | 25.0% | 22.4% |
| 2:0 | 22.6% | 31.9% |
| 3:0 | 18.2% | 26.9% |
| 4:0 | 13.6% | 10.9% |
| 5:0 | **19.3%** | **1.4%** |

### الاحتمالات (مُتحقَّقة على 6000 سحبة)

| الإصدار | معلن | فعلي |
|---|---|---|
| Bronze | 34% | 33.80% |
| Silver | 26% | 26.65% |
| Gold | 18% | 17.87% |
| Elite | 11% | 10.68% |
| Epic | 6% | 5.92% |
| Legendary | 3% | 2.88% |
| Master | 1.5% | 1.47% |
| Super Rare | 0.5% | 0.73% |

الضمان الأرضي يعمل: حزمة ذهبية ترفع Gold من 18% إلى **30.69%** فعلياً
(على 4500 سحبة) لأن البطاقة الأخيرة تُرقّى عند الرول المنخفض.

## التوسع

- كل مباراة ≈ 4KB. خادم 2GB → **500–1000 مباراة متزامنة** بأمان.
- كاش البطاقات (800 صف) مشترك بين كل الغرف: استعلام واحد بدل ألف.
- عند تعدد النسخ: **Redis إلزامي** (`REDIS_URL`) — بدونه لاعبان على
  نسختين مختلفتين لن يلتقيا أبداً.
- نتائج المباريات تُكتب دفعة واحدة، لا صفاً صفاً.
- `SIGTERM` يُنهي المباريات الجارية بدل قطعها.
